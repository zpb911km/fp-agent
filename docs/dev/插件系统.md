# 插件系统 —— 导入、注册与覆写机制（全异步重构版）

> 本文档描述全异步重构后的插件系统架构。插件不是"工具"——它们是 Agent 的神经系统，而非双手。

---

## 1. 设计哲学

**插件的定位**与 Skills/Commands/Tools 有本质区别：

| 维度 | Plugin | Skills / Commands / Tools |
|---|---|---|
| 架构风格 | ABC 基类 + 生命周期钩子注册 | 文件命名约定 + 函数签名约定 |
| 注册方式 | 显式调用 `lifecycle.register()` | 文件扫描 + 自动收集导出函数 |
| 控制反转 | 插件主动注册钩子 → 事件时回调 | 框架按名称查找函数 → 被动调用 |
| 副作用类型 | 横切关注点（通知、审计、过滤、守卫） | 工具执行（查天气、运行代码、搜索） |
| 生命周期影响 | ✅ 可修改/拦截 Agent 主流程 | ❌ 只返回结果，不干涉流程 |
| 卸载机制 | ✅ `on_unregister()` + 钩子追踪清理 | 无 |

插件的核心能力在于**横切关注点注入**——通知、审计、内容过滤、流程守卫等，而非执行具体任务。

## 2. Plugin 抽象基类

位于 `fp_core/plugins/base/plugin.py`：

```python
from abc import ABC, abstractmethod

class Plugin(ABC):
    name: str = "base_plugin"       # 类属性，唯一标识（同名覆写的判定键）
    version: str = "1.0.0"

    @abstractmethod
    def on_register(self, lifecycle: "LifecycleManager"):
        """子类必须实现：在此注册生命周期钩子"""
        ...

    @abstractmethod
    def on_unregister(self):
        """子类必须实现：清理资源"""
        ...

    # 启用/禁用状态控制
    def enable(self):
        self._enabled = True

    def disable(self):
        self._enabled = False

    @property
    def is_enabled(self) -> bool:
        return self._enabled

    # 状态存取
    def get_state(self) -> dict:
        return {"enabled": self._enabled}

    def set_state(self, state: dict):
        self._enabled = state.get("enabled", True)
```

### PluginConfig

```python
@dataclass
class PluginConfig:
    enabled: bool = True
    priority: int = 100
    metadata: dict[str, Any] = field(default_factory=dict)
```

每个插件实例关联一个 `PluginConfig`，但优先级最终由 `lifecycle.register()` 时指定的参数决定。

## 3. PluginRegistry 注册表

`PluginRegistry` 是插件的管理中心，负责扫描、导入、注册和卸载。

```python
class PluginRegistry:
    def __init__(self, lifecycle: LifecycleManager, plugin_dir: str | None = None):
        self._lifecycle = lifecycle
        self._plugins: dict[str, Plugin] = {}          # name → Plugin 实例
        self._plugin_order: list[str] = []              # 保持注册顺序
        self._tracked_hooks: dict[str, list[tuple]] = {}  # 钩子追踪记录
        self._states: dict[str, PluginConfig] = {}       # 插件配置

        if plugin_dir is not None:
            self.scan(plugin_dir)  # 构造时即扫描
```

### 3.1 scan() — 文件命名约定

```python
for entry in os.scandir(plugin_dir):
    if entry.is_dir():            continue  # 跳过子目录
    if not fname.endswith(".py"): continue   # 非 .py 文件
    if fname.startswith("_"):     continue   # 内部模块 _name.py
    if fname in SKIP_FILES:       continue   # base.py, setup.py
```

| 文件名 | 行为 | 说明 |
|---|---|---|
| `notification.py` | ✅ 加载 | 标准插件文件 |
| `notification.py.disabled` | ❌ 跳过 | 后缀不匹配 `.py`，自然过滤 |
| `_notification.py` | ❌ 跳过 | 下划线前缀视为内部模块 |
| `base.py` | ❌ 跳过 | 硬编码忽略（基类模块） |

> **文件即开关**：无需修改代码或配置文件，通过 `mv plugin.py plugin.py.disabled` 即可临时禁用插件，反向操作重新启用。

### 3.2 动态导入

每次 `scan()` 都是全新导入，不缓存模块，理论上支持热重载：

```python
_import_counter = 0  # 类级别计数器

@classmethod
def _import_module(cls, filepath: str):
    cls._import_counter += 1
    module_name = f"_plugin_scan_{cls._import_counter}"
    spec = importlib.util.spec_from_file_location(module_name, filepath)
    module = importlib.util.module_from_spec(spec)
    sys.modules[module_name] = module
    spec.loader.exec_module(module)
    return module
```

### 3.3 Plugin 子类提取 — 三重过滤

从模块中提取 Plugin 子类时，使用三层过滤防止误抓：

```python
for _, obj in inspect.getmembers(module, inspect.isclass):
    if obj is Plugin:                continue  # 跳过基类本身
    if not issubclass(obj, Plugin):  continue  # 非 Plugin 子类
    if obj.__module__ != module.__name__: continue  # 外部导入的类
```

第三重 `__module__` 检查是关键——防止 `from other_plugin import OtherPlugin` 导致的重复注册。

## 4. _TrackingLifecycle 追踪包装器

这是卸载清理的核心设计。插件注册钩子时，所有注册操作都被透明记录，保证卸载时能精确移除。

```mermaid
flowchart LR
    A[PluginRegistry.register] --> B[创建空 tracker 列表]
    B --> C[包装 Lifecycle → _TrackingLifecycle]
    C --> D[调用 plugin.on_register(tracking)]
    D --> E[tracker 记录了所有 register 调用]
    E --> F[保存 tracker → _tracked_hooks[name]]
    F --> G[unregister 时：逐条移除钩子]
```

代码逻辑：

```python
def _register_instance(self, plugin: Plugin):
    tracker: list[tuple] = []
    tracking = _TrackingLifecycle(self._lifecycle, tracker)
    plugin.on_register(tracking)
    self._tracked_hooks[plugin.name] = tracker
```

`_TrackingLifecycle` 包装器将每一次 `register()` 调用都记录到 tracker 中：

```python
class _TrackingLifecycle:
    def __init__(self, lifecycle: LifecycleManager, tracker: list):
        self._lifecycle = lifecycle
        self._tracker = tracker

    def register(self, hook, func, *, priority=100, name="", hook_type="observe"):
        self._lifecycle.register(hook, func, priority=priority, name=name, hook_type=hook_type)
        self._tracker.append((func, hook))  # 记录 (处理函数, 钩子名)
```

卸载时反向清理，实现无侵入、精确、防泄漏的清理：

```python
def unregister(self, name: str):
    if name not in self._plugins:
        return
    plugin = self._plugins.pop(name)
    # 1. 移除所有钩子
    for func, hook_name in self._tracked_hooks.pop(name, []):
        self._lifecycle.unregister(func, hook_name)
    # 2. 调用插件自身清理
    plugin.on_unregister()
```

## 5. 同名覆写机制

同名覆写是插件系统的灵活性关键——用户插件可以完全替换内置插件的行为。

```python
if instance.name in self._plugins:
    old = self._plugins[instance.name]
    self.unregister(old.name)  # 先卸载旧插件的全部钩子
    print(f"[PluginRegistry] 覆盖插件: {instance.name}")
self._register_instance(instance)
```

**完整流程：**
1. 发现同名插件已注册
2. 对旧插件执行 `unregister()`：移除所有钩子 + 调用 `on_unregister()`
3. 新插件的 `on_register()` 被调用，注册新钩子
4. `_plugins[name]` 被更新为新实例

## 6. LifecycleManager 钩子系统

### 6.1 钩子类型

| 类型 | 行为 | 异常处理 |
|---|---|---|
| **transform** | 可修改 `context.data`、守卫/取消流程 | 异常上抛，阻断主流程 |
| **observe** | 只读通知 | 异常隔离，只记日志不干扰主流程 |

`register()` 时的 `hook_type` 参数决定钩子类型：

```python
lifecycle.register(
    LifecycleHook.ON_BEFORE_RESPONSE,
    self.handler,
    priority=50,
    name=self.name,
    hook_type="observe",  # 或 "transform"
)
```

### 6.2 完整钩子列表

| 钩子枚举 | 类型 | 触发时机 | `context.data` 关键字段 |
|---|---|---|---|
| `ON_INIT` | observe | Agent 初始化完成 | — |
| `ON_CONFIG_LOADED` | observe | 配置加载完成 | — |
| `ON_MESSAGE_FILTER` | **transform** | 用户消息过滤/修改 | `content`, `blocked` |
| `ON_MESSAGE_RECEIVED` | observe | 消息已接收 | `content` |
| `ON_BEFORE_LLM_CALL` | **transform** | LLM 调用前 | `messages`, `tools`, `cancelled` |
| `ON_AFTER_LLM_CALL` | **transform** | LLM 返回后 | `response`, `block_tool_execution` |
| `ON_BEFORE_RESPONSE` | **transform** | 返回响应前 | `content`, `modified_content` |
| `ON_TOOL_SELECT` | observe | 工具已选择 | `tools` |
| `ON_TOOL_CALL` | observe | 工具即将调用 | `tool_name`, `tool_args` |
| `ON_TOOL_RESULT` | observe | 工具调用完成 | `tool_name`, `result` |
| `ON_TOOL_ERROR` | observe | 工具错误 | `error` |
| `ON_CONTEXT_UPDATE` | observe | 上下文已更新 | `msg_count` |
| `ON_ERROR` | observe | 发生错误 | `error` |
| `ON_SHUTDOWN` | observe | Agent 关闭中 | — |
| `ON_CLEANUP` | observe | 清理资源 | — |

### 6.3 emit 调用约定

```python
ctx = await lifecycle.emit(LifecycleHook.ON_BEFORE_RESPONSE, content=final_content)
final_content = ctx.data.get("modified_content", final_content)
```

**参数传递机制：**
1. `emit()` 的 `**kwargs` 被合并到 `EmitContext.data` 字典
2. 钩子处理函数通过关键字参数接收这些值
3. **transform** 类型钩子的返回值（修改后的 `context`）会传递给下一个钩子
4. **observe** 类型钩子的返回值被忽略

### 6.4 优先级排序

同一钩子下，多个处理函数按 `priority` 升序执行（小号优先）。

- `priority=10` 的钩子先于 `priority=100` 执行
- 同名钩子间不保证绝对顺序（仅按 priority 分组排序）

## 7. Agent 初始化双层扫描

Agent 启动时依次扫描两个目录，实现"内置 + 用户扩展"的分层架构：

```python
# Step 1 — 内置插件（框架自带）
_builtin_plugin_dir = os.path.join(os.path.dirname(__file__), "..", "plugins")
self.plugins = PluginRegistry(
    self.lifecycle,
    plugin_dir=os.path.normpath(_builtin_plugin_dir),
)

# Step 2 — 用户插件（XDG 约定，同名覆盖）
_xdg_data = os.environ.get(
    "XDG_DATA_HOME",
    os.path.join(os.path.expanduser("~"), ".local", "share"),
)
_user_plugin_dir = os.path.join(_xdg_data, "fp", "plugins")
if os.path.isdir(_user_plugin_dir):
    self.plugins.scan(_user_plugin_dir)
```

**效果：**
- 内置插件定义默认行为
- 用户插件若有同名（`name` 属性相同），会覆盖内置插件
- 用户插件若无同名冲突，则作为额外插件注册

> **当前状态：** 内置 `fp_core/plugins/` 目录目前仅包含 `base/plugin.py`（基类），无实际插件实例。现有的两个插件（`kdeconnect_notifier`、`notification`）均在用户目录 `~/.local/share/fp/plugins/` 中。

## 8. `@hook` 装饰器（半废弃）

```python
def hook(hook_type: LifecycleHook, priority: int = 100):
    def decorator(func):
        func._lifecycle_hook = hook_type
        func._lifecycle_priority = priority
        return func
    return decorator
```

该装饰器仅给函数打上标签（`_lifecycle_hook`、`_lifecycle_priority` 属性），**当前未被框架自动消费**。

**建议**：新插件统一使用手动 `lifecycle.register()` 方式，避免与装饰器模式并存造成混淆。未来可能废弃此装饰器。

## 9. 用户插件编写指南

### 9.1 快速入门

1. 在 `~/.local/share/fp/plugins/` 下创建 `.py` 文件
2. 继承 `Plugin` 类，实现 `on_register()` 和 `on_unregister()`
3. 在 `on_register()` 中调用 `lifecycle.register()` 注册钩子
4. 文件命名不要以 `_` 开头，不要叫 `base.py` 或 `setup.py`
5. 如需禁用插件，加 `.disabled` 后缀（如 `plugin.py.disabled`）

### 9.2 完整示例

```python
"""~/.local/share/fp/plugins/my_plugin.py"""
from fp_core.plugins.base.plugin import Plugin
from fp_core.core.lifecycle import LifecycleHook


class MyPlugin(Plugin):
    name = "my_plugin"
    version = "1.0.0"

    def on_register(self, lifecycle):
        lifecycle.register(
            LifecycleHook.ON_BEFORE_RESPONSE,
            self.on_before_response,
            priority=50,
            name=self.name,
            hook_type="observe",
        )

    def on_unregister(self):
        # 清理资源（打开的文件、网络连接等）
        pass

    async def on_before_response(self, context, **kwargs):
        content = context.data.get("content", "")
        print(f"[MyPlugin] 回复摘要: {content[:50]}")
        return context
```

### 9.3 编写 transform 类型钩子（修改/拦截流程）

```python
class GuardPlugin(Plugin):
    name = "content_guard"
    version = "1.0.0"

    def on_register(self, lifecycle):
        # 注册为 transform 类型，可修改或阻断流程
        lifecycle.register(
            LifecycleHook.ON_MESSAGE_FILTER,
            self.filter_message,
            priority=10,
            name=self.name,
            hook_type="transform",
        )

    def on_unregister(self):
        pass

    async def filter_message(self, context, **kwargs):
        content = context.data.get("content", "")
        blocked_words = ["secret_token_123"]

        for word in blocked_words:
            if word in content:
                context.data["blocked"] = True
                context.data["content"] = "[消息已被内容过滤器拦截]"
                break

        return context
```

## 10. 架构总评

### 10.1 亮点

| 设计 | 评价 |
|---|---|
| **`_TrackingLifecycle` 追踪包装器** | 无侵入追踪注册记录，完美支撑卸载清理，防止钩子泄漏 |
| **三重过滤 + 动态导入** | 模块隔离稳健，避免误抓外来类；每次 scan() 全新加载，支持热重载 |
| **Transform/Observe 双模钩子** | 兼顾流程修改（transform）与通知观察（observe），异常处理策略分离 |
| **文件即开关机制** | `.py` / `.py.disabled` 切换，无需配置文件即可启停插件 |
| **同名覆写 + 双层扫描** | 用户插件可通过同名覆盖内置插件，同时保留 XDG 标准化路径 |

### 10.2 注意点

| 问题 | 说明 |
|---|---|
| **`@hook` 装饰器半废弃** | 与手动 `lifecycle.register()` 并存，可能造成混淆。建议统一使用手动注册方式。 |
| **内置 plugins/ 目录为空** | 当前 `fp_core/plugins/` 仅含基类，无实际内置插件。双层扫描的"内置层"实际无效果。 |
| **scan() 无增量/缓存** | 每次 scan() 重新扫描整个目录并全部导入，大目录下存在性能隐患。 |
| **热重载未完全验证** | 虽然动态导入不缓存模块，但已注册的钩子和插件实例的替换逻辑未在真实场景中充分测试。 |
