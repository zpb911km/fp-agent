# 子代理派遣（subagent）— 战略指南

## 核心原理

每次 LLM 调用花的是 token，token 花在输入（context）和输出（generation）上。
**主上下文的每一轮都在膨胀**——添加用户消息、工具结果、我的回复。越膨胀越慢越贵。

subagent 的解法：**把"共享内存"模型改成"函数调用栈"模型**。

```
传统方式（所有信息在主上下文里）：
  ┌─────────────────────────────────────┐
  │ system(2K) + 历史(5K) + 工具结果(3K) │ → 每次请求 prefix 都变 → 缓存永不命中
  └─────────────────────────────────────┘

subagent 方式（子任务独立执行）：
  主上下文: system(2K) + 规划(0.5K)          → prefix 稳定 ✓
  子代理 A:  system(0.3K) + 任务(0.2K) + 工具结果(1K)  → prefix 稳定 ✓ → 缓存命中
  子代理 B:  system(0.3K) + 任务(0.2K) + 工具结果(1K)  → prefix 稳定 ✓ → 缓存命中
```

## Token 节省公式

对于一次包含 N 次工具调用、耗时 M 轮的复杂任务：

| 指标 | 传统单线程 | 子代理派遣 | 节省 |
|:----|:----------:|:----------:|:----:|
| 主上下文膨胀 | O(N×工具结果) | O(1) | 60-80% |
| API 缓存命中 | ❌ 0%（prefix 每轮变） | ✅ 高（task prefix 不变） | 50% |
| 注意力计算 | O(n²) | O(k²)+O(m²)，k,m<<n | 可观 |

**经验法则**：任何需要 **≥2 次工具调用** 或 **读取大文件** 的任务，都用 subagent。

## 使用场景清单

### ✅ 强烈推荐使用 subagent

| 场景 | 原因 | 替代方案的问题 |
|------|------|--------------|
| 读取/分析大文件（代码、日志、数据） | 文件内容不出主上下文 | 直接 read_file → 几千 token 灌进主上下文 |
| 多步数据分析（抓取 → 解析 → 整理） | 中间结果不污染主上下文 | 每步都在主上下文膨胀 |
| 批量文件处理 | 循环在子代理内完成 | 工具调用炸掉上下文 |
| 代码调试（读源码 → 复现 → 修复） | 完整推理链在局部 | 所有临时结论留在主上下文 |
| 信息检索与综合（搜索 → 阅读 → 总结） | 只返回摘要 | 每篇网页全文占用 token |

### ❌ 不需要 subagent

| 场景 | 原因 |
|------|------|
| 单步查询（`python_syntax_check` 检查一个文件） | 一步完成，子代理反而有启动开销 |
| 用户简单对话 | 不需要额外上下文隔离 |
| 工具结果小于 200 token | 启动子进程开销 > 节省 |

## 使用技巧

1. **task 要自包含**：子代理没有当前对话上下文，所有背景信息通过 `context` 参数传递
2. **利用 `store_result`**：子代理结果存入记忆后，主代理可随时通过 `memory_read` 访问
3. **善用 `constraints.verbose=false`**（默认）：子代理的完整推理链条不进入主上下文，只返回结论
4. **复杂分析用 `output_format=json`**：结构化输出便于主代理解析使用
5. **先预估再派**：如果任务简单到 1-2 步能完成，自己做；否则无脑丢给 subagent
