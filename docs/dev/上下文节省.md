# Shortcircuit（短路）— 命令规范 V3

> V3 核心变更：去除 `_shortcircuited` 标记，改用**隐式状态检测**——连通块中只有 user + assistant 两条消息即为"已充分压缩"，超过两条则有继续短路空间。

---

## 1. 核心概念

### 1.1 连通块（Component）

> 对应 V1 中的"交互"，V2 中更名为连通块，V3 沿用。

一个**连通块**是 `_messages` 中一个语义连续的已完成交互单元，其边界定义为：

```
从一条 user 消息开始
到第一条 role == "assistant" 且无 tool_calls 的消息结束
中间可以包含零或多条 tool_call/tool/assistant(有tc) 消息
```

### 1.2 连通块的两种形态

| 形态 | 结构 | 是否可再短路 |
|------|------|-------------|
| **充分压缩态** | `user + assistant(无tc)` 恰好 2 条 | `--crop` 无效果；`--regenerate` 可进一步提炼 |
| **可压缩态** | 包含 tool_call/tool/多次 assistant 等中间消息，> 2 条 | 是，crop/regenerate 均有效 |

两者之间**无标记区分**，完全由消息数量和角色隐式判定。

### 1.3 短路（Shortcircuit）

将**可压缩态的连通块**压缩至**充分压缩态**：

- **crop 模式**（裁剪）：直接移除 tool_call/tool 等中间消息，保留 user 原文和 assistant 终回复原文，变为恰好 2 条
- **regenerate 模式**（提炼）：在 crop 基础上，再将 assistant 终回复传入 LLM 精炼为更简洁的版本

对于已经是**充分压缩态**的连通块，`crop` 无效果（已经是最简），`regenerate` 仍可调用 LLM 重新提炼回复。

## 2. 命令设计

### 2.1 命令名

| 字段 | 值 |
|------|-----|
| `name` | `shortcircuit` |
| `aliases` | `["sc"]` |

### 2.2 参数语法

```
/sc                    → 短路最近 1 个可压缩态的连通块
/sc N                  → 短路最近 N 个可压缩态的连通块
/sc list               → 显示所有连通块概览（含充分压缩态标记）
/sc #N                 → 短路编号为 N 的连通块（不限形态）
/sc #M-#N              → 短路编号 M 到 N 的连通块（闭区间）

可选修饰：
  -c                   → 裁剪模式（crop）：只移除 tool 中间消息，不调 LLM
  -r                   → 提炼模式（regenerate）：调 LLM 重新生成精简回复（默认）
```

**语义要点**：

- `/sc` 和 `/sc N` 默认只作用于**可压缩态**连通块（> 2 条消息的），充分压缩态的自动跳过
- `/sc #N` 和 `/sc #M-#N` 不限形态，选中的全部处理——充分压缩态的走 regenerate（如果 mode 是 regenerate）或跳过（如果 mode 是 crop）
- 一个连通块可以被多次短路（regenerate 模式），每次都会再次提炼

### 2.3 连通块编号与展示

**编号规则**：1-based，从最早到最新，从左往右编号。

```
连通块列表（共 4 个，~ = 已充分压缩）:

  #1  ~ [用户] 帮我查天气
        [AI]   北京25°C...                          ← 充分压缩态（2条）
  
  #2    [用户] 哪个适合户外运动？
        [AI]   北京更适合                            ← 充分压缩态（2条）
  
  #3    [用户] 帮我写一个 python 脚本，处理 CSV...
        [AI]   [tool_call: read_file]               ← 可压缩态（>2条）
        [工具] 文件内容...
        [AI]   [tool_call: write_file]
        [工具] 写入成功
        [AI]   已生成脚本，保存到 output.py
  
  #4    [用户] 帮我把注释改成中文                   ← 当前（最近一个，也是可压缩态）
        [AI]   [tool_call: read_file]
        [工具] 文件内容...
        [AI]   已修改完成
```

| 命令 | 行为 |
|------|------|
| `/sc` | 短路 #4（最近的可压缩态连通块）|
| `/sc 2` | 短路 #3、#4（最近 2 个可压缩态）|
| `/sc #2` | 短路 #2（充分压缩态，regenerate 下会重新提炼）|
| `/sc #1-#2 -c` | 短路 #1、#2（crop 模式下无效果——已是最简）|
| `/sc list` | 显示上述列表 |

## 3. 连通块检测算法

### 3.1 核心检测：`_scan_components()`

从尾部向前扫描，识别所有连通块。

```python
def _scan_components(self) -> list[dict]:
    """
    扫描 _messages，返回从旧到新排序的连通块列表。
    
    每条记录：
    {
        "idx": 1,                   # 连通块编号（1-based）
        "user_idx": 1,              # _messages 中的索引
        "terminal_idx": 2,          # _messages 中的索引（终点 assistant）
        "message_count": 2,         # 该连通块包含的消息总数
        "user_preview": "帮我查天气",
        "assistant_preview": "北京25°C...",
        "compressible": True/False, # True = 消息数 > 2，可压缩
    }
    """
```

### 3.2 检测规则

```
从 _messages 尾部向前遍历（i = len-1 → 1，跳过 [0] system）:

Case A: i 是 assistant(无 tc)
  ↓ 这是一个连通块的终点
  向前找最近的 user（跳过中间所有非 user 消息）
    ↓ 找到 user → 连通块边界 = (user_idx, i)
    计算 message_count = i - user_idx + 1
    compressible = (message_count > 2)
    i = user_idx - 1（跳过该连通块，继续往前）

Case B: i 是 user 且之前没有 assistant(无 tc) 与之配对
  ↓ 此用户消息处于"活跃"状态（交互未完成）
  停止扫描，不纳入连通块列表
```

**为什么尾部向前？** 默认 `/sc` 短路最近的连通块，反向扫描产出顺序天然从近到远，取第一个可压缩态即可。

### 3.3 隐式充分压缩态检测

不需要任何标记字段。一个连通块若 `message_count == 2` 且结构为 `user + assistant(无tc)`，即自动判定为充分压缩态——这就是"短路后"应有的样子。

```
充分压缩态 = compressible == False
可压缩态   = compressible == True
```

## 4. 核心数据操作：`ConversationState.shortcircuit()`

### 4.1 签名

```python
async def shortcircuit(
    self,
    refiner: Callable[[str, str], Awaitable[tuple[str, str]]] | None,
    indices: list[tuple[int, int]],  # [(user_idx, terminal_idx), ...]
    mode: str = "regenerate",        # "regenerate" | "crop"
) -> tuple[bool, str, int]:
    """
    Args:
        refiner:  提炼回调，(user_text, assistant_text) → (new_user, new_assistant)
                  crop 模式传 None
        indices:  要短路的连通块消息索引（按时间顺序传入）
        mode:     操作模式
        
    Returns:
        (是否成功, 描述信息, 压缩减少的消息数)
    """
```

### 4.2 三步执行 + 快照回滚

```
输入: indices = [(1,2), (3,7)]  # 两个连通块

Step 1 — 快照
─────────────────────────────────────────
snapshot = list(self._messages)

Step 2 — 逐块处理
─────────────────────────────────────────
new_sections = []         # 存放重建后的各个段
total_saved = 0

for user_idx, terminal_idx in indices:
    user_msg = self._messages[user_idx]
    terminal_msg = self._messages[terminal_idx]
    msg_count = terminal_idx - user_idx + 1
    
    if msg_count == 2:
        # 已处于充分压缩态
        if mode == "crop":
            # crop 无效果，原样保留
            new_sections.append([user_msg, terminal_msg])
        else:  # regenerate
            # 仍可重新提炼 assistant 回复
            new_user, new_assistant = await refiner(
                user_msg["content"], terminal_msg["content"]
            )
            new_sections.append([
                {"role": "user", "content": new_user},
                {"role": "assistant", "content": new_assistant},
            ])
            # 消息数不变，但 assistant 内容被精炼
    else:
        # 可压缩态
        if mode == "crop" or refiner is None:
            # 裁剪模式：只保留 user + 终回复
            new_sections.append([
                {"role": "user", "content": user_msg["content"]},
                {"role": "assistant", "content": terminal_msg["content"]},
            ])
        else:
            # 提炼模式：裁剪 + LLM 精炼终回复
            new_user, new_assistant = await refiner(
                user_msg["content"], terminal_msg["content"]
            )
            new_sections.append([
                {"role": "user", "content": new_user},
                {"role": "assistant", "content": new_assistant},
            ])
        total_saved += msg_count - 2
    
    # ← 提炼全部成功才继续
    # ← 任一失败 → self._messages = snapshot → return (False, 原因, 0)

Step 3 — 一次性重建
─────────────────────────────────────────
new_messages = [self._messages[0]]  # system
i = 1
section_idx = 0

while i < len(self._messages):
    if (section_idx < len(indices) 
        and i == indices[section_idx][0]):
        # 命中短路块 → 追加压缩后的 section
        for msg in new_sections[section_idx]:
            new_messages.append(dict(msg))
        i = indices[section_idx][1] + 1
        section_idx += 1
    else:
        new_messages.append(dict(self._messages[i]))
        i += 1

self._messages = new_messages
```

### 4.3 原子性

```
提炼全部成功 → 一次性重建 → 成功返回
任一提炼失败 → snapshot 回滚 → 失败返回
重建阶段异常 → snapshot 回滚 → 失败返回
```

所有 LLM 调用（IO）完成后才修改 `_messages`，无中间态暴露。

## 5. Agent 编排层

### 5.1 新增方法

```python
# agent.py

async def shortcircuit_context(
    self, count: int = 1, indices: list[int] | None = None, mode: str = "regenerate"
) -> dict:
    """短路上下文（公共 API）

    Args:
        count:   短路最近 N 个可压缩态的连通块
        indices: 短路指定编号的连通块（不限形态）
        mode:    "regenerate" | "crop"

    Returns:
        {"ok": bool, "msg": str, "saved": int, "count": int}
    """
```

### 5.2 流程

```
shortcircuit_context():
  1. 调用 _conv.scan_components() → components[]
  
  2. 如果指定了 indices（/#N 或 #M-#N）:
      从 components 中按 idx 筛选目标连通块
    否则（/sc 或 /sc N）:
      从最近的 components 中选 N 个 compressible == True 的
  
  3. 如果无可处理的连通块 → 返回 {"ok": False, "msg": "...", ...}
  
  4. 定义 refiner:
    if mode == "crop":
        refiner = None
    else:
        refiner = self._build_regenerate_refiner()
  
  5. 调用 _conv.shortcircuit(refiner, raw_indices, mode)
     → (success, msg, saved)
  
  6. if success:
      self.session.save_context(self._conv.messages)
      return {"ok": True, "msg": ..., "saved": int, "count": int}
    else:
      return {"ok": False, ...}
  
  7. 所有 display 由命令层 (commands/shortcircuit.py) 负责
```

### 5.3 refiner 实现

```python
async def _build_regenerate_refiner(self):
    """构建提炼回调"""
    async def refiner(user_text: str, assistant_text: str) -> tuple[str, str]:
        prompt = (
            "请将以下对话压缩为一对精简的对话消息。\n\n"
            "原始对话：\n"
            f"用户：{user_text}\n"
            f"AI：{assistant_text}\n\n"
            "要求：\n"
            "1. 保留用户消息原文，不做任何修改\n"
            "2. 将 AI 回复精简为 1-3 句话，保留关键信息\n"
            "3. 去掉工具调用细节、中间步骤、语气词\n"
            "4. 只输出以下格式：\n"
            "   用户：{用户原文}\n"
            "   AI：{精简后的 AI 回复}"
        )
        result = await self._llm.summarize(
            f"用户：{user_text}\nAI：{assistant_text}",
            instruction=prompt,
        )
        # 解析 result，提取 "用户：" 和 "AI：" 行
        ...
    return refiner
```

## 6. 断链防御

### 6.1 原理

`/sc list` 的每一行显示的是一个完整的连通块（user + assistant 预览），而不是单条消息。

```
✅ 连通块级别的展示（用户选择 #2 或范围 #1-#3，始终是完整语义单元）:

  #1  ~ [用户] 帮我查天气           (2条)
         [AI]   北京25°C...

  #2    [用户] 帮我写脚本           (5条)
         [AI]   已生成...

  #3    [用户] 加注释              (3条)
         [AI]   已添加...

❌ 单条消息级别的展示（用户可能选中 tool 消息导致断链）:

  [1] 用户: 帮我查天气
  [2] AI:   ...                    ← 危险选择
  [3] 工具: {...}
  [4] AI:   北京25°C...
```

用户对连通块的操作粒度只有"整个块"——通过编号（#N）或范围（#M-#N）。不存在选中块内某条消息导致上下文结构被破坏的可能。

### 6.2 为什么 #N 引用的是连通块编号而非消息索引？

因为连通块编号天然稳定。消息索引会因前后的短路操作而漂移，但连通块的"第 3 个已完成交互"这个语义是稳定的。在显示 list 时标出的编号就是用户引用的目标。

## 8. 完整用例

### 场景 1：用户刚完成一次工具调用，想裁剪

```
用户: "画一张北京地图"
→ AI: [tool_call: map_api]
→ 工具: {数据}
→ AI: [tool_call: annotate_api]
→ 工具: {标注}
→ AI: "已生成，故宫位于中轴线"

用户: /sc
→ 将 #3（最近的可压缩态连通块）裁剪为：
  用户: "画一张北京地图"
  AI: "已生成，故宫位于中轴线"
→ 输出: "✅ 已压缩 #3：5 条 → 2 条（节省 3 条）"
```

### 场景 2：用户觉得已经裁剪过的回复不够精炼

```
用户: /sc #1 -r
→ #1 已是充分压缩态（2条），但 regenerate 模式会重新提炼
→ LLM 将 AI 回复从"已生成北京地图，故宫位于中轴线中心" 精炼为 "已生成，故宫居中"
→ 输出: "✅ 已重新提炼 #1"
```

### 场景 3：用户想批量裁剪最近的 3 个连通块

```
用户: /sc 3
→ 找到最近的 3 个可压缩态连通块（#5, #4, #3）
→ 全部裁剪后重建
→ 输出: "✅ 已压缩 3 个连通块，共节省 12 条消息"
```

### 场景 4：用户查看有哪些连通块

```
用户: /sc list

📦 连通块列表（共 5 个，~ = 已充分压缩）:

  #1  ~ [用户] 帮我查天气               (2条)
         [AI]   北京25°C...

  #2  ~ [用户] 哪个适合户外运动？       (2条)
         [AI]   北京更适合

  #3    [用户] 帮我写脚本               (5条)
         [AI]   已生成...

  #4    [用户] 加注释                  (3条)
         [AI]   已添加...

  #5    [用户] 帮我测试                 (4条)
         [AI]   已测试完成...
```
