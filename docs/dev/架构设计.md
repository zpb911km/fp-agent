## 分层架构

Five Pebbles 采用**五层架构**设计，从入口到底层数据逐层收敛，职责清晰。采用 monorepo 多包管理，核心引擎与界面层完全解耦。

```

│                     路由入口层 (Router)                           │
│                                                                  │
│   packages/fp/src/fp/main.py                                     │

│   │  argparse → --mode cli|webui|acp → 分发到对应界面包      │   │
│   └──────────────────────────┬───────────────────────────────┘   │
└──────────────────────────────┼───────────────────────────────────┘

                               ▼
```

**各层职责**：

| 层 | 包/模块 | 职责 |
|----|---------|------|
| 路由入口 | `fp` | `--mode` 参数分发，选择界面层启动模式 |
| 界面层 | `fp-cli`, `fp-webui`, `fp-acp` | 终端/Web/API 交互，通过 IOChannel 与核心通信 |
| 核心层 | `fp-core` | Agent 主循环编排，LLM 调用，生命周期管理，会话管理 |
| 插件/工具层 | `fp-core` 内的 tools/plugins/skills/commands | 工具注册与执行，生命周期钩子挂载，技能热加载 |
| 数据层 | 文件系统（XDG 标准路径） | 会话持久化、长期记忆、任务追踪、配置管理 |

## IO 通道体系

IOChannel 是一个抽象协议层，将界面层与核心层解耦。`Agent` 和命令内部通过 `IOChannel` 接口与用户交互，不直接依赖 `input()` 或 `display` 模块。

```
                    ┌──────────────────┐
                    │    Agent / 命令   │
                    │    (业务逻辑)      │
                    └────────┬─────────┘
                             │ 通过 IOChannel 交互
                             ▼
                    ┌──────────────────┐
                    │    IOChannel     │  ← 抽象基类
                    │   (core/io.py)   │
                    └────────┬─────────┘
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
        ┌──────────┐  ┌────────────┐  ┌──────────┐
        │  CLIIO   │  │WebSocketIO │  │  RestIO  │
        │          │  │            │  │          │
        │ input()  │  │ EventBus   │  │ ask() →  │
        │ display  │  │ + Queue    │  │ 返回 ""  │
        │ 着色输出  │  │ 异步推送    │  │ 静默模式  │
        └──────────┘  └────────────┘  └──────────┘
              │              │              │
              ▼              ▼              ▼
          fp-cli         fp-webui        fp-acp
```

**接口定义**：

```python
class IOChannel:
    async def ask(self, prompt: str) -> str: ...   # 提问，获取回复
    def say(self, text: str): ...                   # 输出普通信息
    def info(self, text: str): ...                  # 信息（绿色）
    def hint(self, text: str): ...                  # 提示（灰色）
    def error(self, text: str): ...                 # 错误（红色）
    def item(self, text: str): ...                  # 列表项（灰色缩进）
```

**三种实现**：

| 实现 | 适用界面 | ask() 行为 | 输出行为 |
|------|---------|-----------|---------|
| `CLIIO` | fp-cli | 阻塞等待终端输入 | display 模块着色输出 |
| `WebSocketIO` | fp-webui | 发布 "ask" 事件到 EventBus，等待 `feed_reply()` 注入 | EventBus 推送 "say"/"info"/"error" 事件 |
| `RestIO` | fp-acp | 直接返回 `""`（无交互能力） | 静默丢弃所有输出 |

## 数据流

一次用户输入的完整处理链路：

```
用户输入 "查询今天的天气"
       │
       ▼
┌───────────────────────────────────────┐
│ 1. 界面层接收                          │
│    CLIIO.ask() / WebSocket 消息 /      │
│    REST 请求                           │
└──────────┬────────────────────────────┘
           │
           ▼
┌───────────────────────────────────────┐
│ 2. 命令判断 (Agent.process())          │
│    ┌─────────────────────────────┐    │
│    │ 以 / 开头 → handle_command  │    │
│    │ 否则 → _process_inner()    │    │
│    └─────────────────────────────┘    │
└──────────┬────────────────────────────┘
           │ (非命令)
           ▼
┌───────────────────────────────────────┐
│ 3. 生命周期: ON_MESSAGE_FILTER        │
│    【transform】可修改/阻止消息        │
│    上下文: MessageFilterEvent          │
└──────────┬────────────────────────────┘
           ▼
┌───────────────────────────────────────┐
│ 4. LLM 调用循环 (while)               │
│    ┌─────────────────────────────┐    │
│    │ 构建上下文 (system + 历史)  │    │
│    │         │                   │    │
│    │ 生命周期: ON_BEFORE_LLM     │    │
│    │ 【transform】可修改 messages │    │
│    │ 上下文: BeforeLLMCallEvent  │    │
│    │         │                   │    │
│    │ LLMClient.chat.completions  │    │
│    │ .create() → Response        │    │
│    │         │                   │    │
│    │ 生命周期: ON_AFTER_LLM      │    │
│    │ 【transform】可修改 response │    │
│    │ 上下文: AfterLLMCallEvent   │    │
│    │         │                   │    │
│    │ ┌── 有 tool_calls? ───┐   │    │
│    │ │ 是          │ 否     │   │    │
│    │ ▼             ▼       │   │    │
│    │ 工具执行     返回内容  │   │    │
│    │ (ToolExec)   退出循环 ┘   │    │
│    │    │                     │    │
│    │ 生命周期: ON_TOOL_*     │    │
│    │    │                     │    │
│    │ 结果追加到上下文,继续    │    │
│    └─────────────────────────────┘    │
└──────────┬────────────────────────────┘
           ▼
┌───────────────────────────────────────┐
│ 5. 上下文裁剪 (context pruning)       │
│    超过 MAX_CONTEXT_TOKENS 时          │
│    移除最早的非 system 消息            │
└──────────┬────────────────────────────┘
           ▼
┌───────────────────────────────────────┐
│ 6. 生命周期: ON_BEFORE_RESPONSE       │
│    【transform】可修改最终回复         │
│    上下文: BeforeResponseEvent          │
└──────────┬────────────────────────────┘
           ▼
┌───────────────────────────────────────┐
│ 7. 持久化                              │
│    session.save_context()             │
│    session.save_message()             │
│    更新文件内嵌 meta                  │
└──────────┬────────────────────────────┘
           ▼
┌───────────────────────────────────────┐
│ 8. 通过 IOChannel 输出到界面           │
│    io.say() / EventBus 推送 / REST    │
└───────────────────────────────────────┘
```

**关键设计点**：

- **LLM 调用是 while 循环**。每次迭代检查返回的 `tool_calls`，有工具调用则执行、结果追加到上下文、继续；无工具调用则退出。
- **IO 通道解耦**。核心层只依赖 `IOChannel` 抽象，不关心具体界面。
- **生命周期钩子**在每一步关键节点被触发，插件可 observe 或 transform 数据流。
- **上下文裁剪**是避免 token 爆炸的最后防线。

## 生命周期系统

### LifecycleManager

`fp_core/core/lifecycle.py` 中的 `LifecycleManager` 实现了事件驱动的钩子系统，是整个插件体系的基石。

### 14 个钩子 (3 transform + 11 observe)

#### 初始化阶段

| 钩子 | 类型 | 触发时机 |
|------|------|----------|
| `ON_INIT` | observe | Agent 初始化完成 |
| `ON_CONFIG_LOADED` | observe | 配置加载完成 |

#### 消息处理阶段

| 钩子 | 类型 | 触发时机 | 事件上下文 |
|------|------|----------|-----------|
| `ON_MESSAGE_FILTER` | **transform** | 用户消息过滤/修改 | `MessageFilterEvent` |
| `ON_MESSAGE_RECEIVED` | observe | 消息已接收（仅通知） | — |

#### 执行阶段

| 钩子 | 类型 | 触发时机 | 事件上下文 |
|------|------|----------|-----------|
| `ON_BEFORE_LLM_CALL` | **transform** | LLM 调用前 — 可修改 messages/tools，或取消调用 | `BeforeLLMCallEvent` |
| `ON_AFTER_LLM_CALL` | **transform** | LLM 返回后 — 可修改 response，或阻止工具执行 | `AfterLLMCallEvent` |

#### 响应阶段

| 钩子 | 类型 | 触发时机 | 事件上下文 |
|------|------|----------|-----------|
| `ON_BEFORE_RESPONSE` | **transform** | 返回响应前 — 可修改 content | `BeforeResponseEvent` |

#### 工具阶段

| 钩子 | 类型 | 触发时机 | 事件上下文 |
|------|------|----------|-----------|
| `ON_TOOL_SELECT` | observe | 工具已选择 | — |
| `ON_TOOL_CALL` | observe | 工具即将调用 | `ToolCallEvent` |
| `ON_TOOL_RESULT` | observe | 工具调用完成 | `ToolResultEvent` |
| `ON_TOOL_ERROR` | observe | 工具错误 | — |

#### 上下文阶段

| 钩子 | 类型 | 触发时机 | 事件上下文 |
|------|------|----------|-----------|
| `ON_CONTEXT_UPDATE` | observe | 上下文已更新 | `ContextUpdateEvent` |

#### 错误阶段

| 钩子 | 类型 | 触发时机 | 事件上下文 |
|------|------|----------|-----------|
| `ON_ERROR` | observe | 发生错误 | `ErrorEvent` |

#### 资源阶段

| 钩子 | 类型 | 触发时机 |
|------|------|----------|
| `ON_SHUTDOWN` | observe | 关闭中 |
| `ON_CLEANUP` | observe | 清理资源 |

### HookContext 传递机制

```python
@dataclass
class HookContext:
    hook: LifecycleHook          # 当前钩子枚举
    data: dict                   # 事件负载数据
    metadata: dict               # 元数据
    stop_propagation: bool       # 停止后续钩子传播
    error: Exception | None      # 执行错误
    hook_type: str               # "observe" | "transform"
```

**传递规则**：

1. **priority 升序执行** — 优先级数值越小越先执行（priority=0 最优先）
2. **stop_propagation** — 设为 `True` 后停止后续钩子
3. **transform 型钩子** — 异常会向上传播（抛出），导致流程中断
4. **observe 型钩子** — 异常被隔离捕获，不中断流程

### Typed Event Context

每个 transform 型钩子有明确的 typed event context，使用 `@dataclass` 定义输入/输出字段：

| Event Context | 关联钩子 | 关键字段 |
|--------------|---------|---------|
| `MessageFilterEvent` | `ON_MESSAGE_FILTER` | `original_content`, `filtered_content`, `blocked` |
| `BeforeLLMCallEvent` | `ON_BEFORE_LLM_CALL` | `messages`, `tools`, `cancelled` |
| `AfterLLMCallEvent` | `ON_AFTER_LLM_CALL` | `response`, `has_tool_calls`, `block_tool_execution` |
| `BeforeResponseEvent` | `ON_BEFORE_RESPONSE` | `content`, `modified_content` |

### 同步/异步自动适配

```python
# 钩子函数可以是同步或异步，emit() 自动检测
if asyncio.iscoroutinefunction(func):
    result = await func(context, **context.data)
else:
    result = func(context, **context.data)
```

### API

```python
# 注册钩子
lifecycle.register(
    hook=LifecycleHook.ON_BEFORE_LLM_CALL,
    func=my_hook_handler,
    priority=100,         # 越小越优先
    name="my_handler",    # 可选，用于 unregister
)

# 注销钩子
lifecycle.unregister(LifecycleHook.ON_BEFORE_LLM_CALL, "my_handler")

# 异步触发
context = await lifecycle.emit(LifecycleHook.ON_BEFORE_LLM_CALL, context)

# 同步触发（非异步环境）
context = lifecycle.emit_sync(LifecycleHook.ON_SHUTDOWN, context)
```

## 配置与数据路径

### 配置系统

采用**三级优先级**（高 → 低）：
1. **环境变量** — 运行时覆盖（最高优先级）
2. **用户配置文件** — `$XDG_CONFIG_HOME/fp/config.json`
3. **硬编码默认值** — Python 模块级常量（最低优先级）

关键配置项：

| 键 | 类型 | 说明 |
|----|------|------|
| `LLM_API_KEY` | str | API 密钥（必填） |
| `LLM_API_BASE_URL` | str | API 基础地址（必填） |
| `LLM_MODEL` | str | 模型名称（必填） |
| `TEMPERATURE` | float | 生成温度 [0.0, 2.0] |
| `MAX_TOKENS` | int | 最大输出 token 数 |
| `TIMEOUT` | int | 请求超时秒数 |
| `RETRY_COUNT` | int | 失败重试次数 |
| `MAX_ITERATIONS` | int | 最大迭代轮次 |
| `MAX_CONTEXT_TOKENS` | int | 上下文窗口 token 数 |
| `MEMORY_MAX_HISTORY` | int | 记忆保留历史条数 |

### 数据存储（XDG 标准）

所有持久化数据遵循 XDG Base Directory 规范：

```
用户配置:  $XDG_CONFIG_HOME/fp/config.json
          （默认 ~/.config/fp/config.json）

会话数据:  $XDG_DATA_HOME/fp/sessions/
记忆数据:  $XDG_DATA_HOME/fp/memory/
任务数据:  $XDG_DATA_HOME/fp/tasks.json
          （默认 ~/.local/share/fp/...）
```

## 架构演进对照

| 维度 | 旧架构 (v1) | 新架构 (v2) |
|------|------------|------------|
| 层数 | 四层 | 五层（拆分路由入口 + 界面层） |
| 包结构 | 单体 | monorepo（5 个包） |
| 界面 | CLI only | CLI / WebUI / ACP 三模式 |
| IO 耦合 | 直接 input/display | IOChannel 抽象解耦 |
| 生命周期钩子 | 19 个 | 14 个（3 transform + 11 observe） |
| 数据路径 | 项目内 data/ | XDG 标准路径 |
| 配置 | 单层默认值 | 三级优先级（环境变量 > 文件 > 默认值） |
